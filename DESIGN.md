# Homework 1.1 设计文档 - 数独领域对象接入方案

## A. 领域对象如何被消费

### 1. 消费层级结构
本项目采用了 **Store Adapter (适配器模式)**。View 层（Svelte 组件）并不直接持有 `Sudoku` 或 `Game` 的原始实例，而是通过 `gameStore` 进行间接消费。

* **View 层消费对象**：组件直接订阅的是 `gameStore`。
* **获取的数据**：通过 `$gameStore` 语法，组件可以实时获取最新的 `grid` (当前盘面)、`initialGrid` (原始题目盘面)、以及 `canUndo`/`canRedo` (撤销/重做可用状态)。
* **操作入口**：
    * **填字**：调用 `gameStore.guess(x, y, value)`。
    * **历史导航**：调用 `gameStore.undo()` 或 `gameStore.redo()`。
    * **新游戏**：调用 `gameStore.start()`，该方法内部实现了 5 题随机抽取逻辑。

### 2. 领域对象的变化反馈
当用户触发操作时，`gameStore` 内部的 `Game` 领域实例会执行逻辑。操作完成后，Store 会执行 `refresh()` 函数，通过 `set()` 方法向所有订阅者发送一个新的状态快照。

## B. 响应式机制说明

### 1. 依赖机制
本项目深度依赖 Svelte 3 的 **Writable Store** 机制。

### 2. UI 更新原理
* **显式赋值触发**：由于 JavaScript 无法自动感知对象深层属性的变化，我们遵循 Svelte 的“赋值即响应”原则。在 `gameStore` 的 `refresh` 方法中，我们每次都创建一个全新的对象字面量传给 `set()`。
* **订阅机制**：组件中使用 `$gameStore` 自动管理订阅与取消订阅。每当 `set()` 被调用，Svelte 编译器生成的代码会感知到引用的变化，从而标记相关 DOM 节点为 dirty 并触发重绘。

### 3. 直接 Mutate 的风险
如果在领域对象内部直接修改二维数组而不触发 Store 的 `set`，Svelte 的响应式系统将无法捕获到内存地址未变的变化，导致 UI 与数据状态脱节（即数据已变但界面不刷新）。

## C. 改进说明

### 1. 相比 HW1 的实质性改进
* **职责边界清晰化**：在 HW1 中，`Sudoku` 只是简单的数据容器。在 HW1.1 中，`Sudoku` 承担了**核心业务约束**——它持有了 `initialGrid` 并能在 `guess` 方法中拦截对初始题目的非法修改，真正体现了领域模型的“自守卫”特性。
* **接入适配层**：引入了 `createGameStore`。HW1 的领域对象是孤立的纯 JS 类，而现在的适配层解决了“纯业务代码”与“UI 框架响应式特性”之间的鸿沟。
* **状态外表化**：改进了 `toJSON` 和 `refresh` 逻辑，确保 UI 能感知到撤销/重做的边界状态。

### 2. 设计取舍 (Trade-off)
* **快照策略**：目前的 Undo 采用了全量克隆 `Sudoku` 对象的快照策略。
    * **优点**：实现简单，逻辑稳健，撤销速度极快。
    * **缺点**：对于极大规模的历史记录会占用较多内存。但考虑到数独 9x9 的体量，这是性能与复杂度的最优平衡点。
